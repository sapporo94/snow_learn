Tables, records, ACLs, and dictionary basics

This is the foundation page for understanding what a ServiceNow table is, how records flow through it, how fields are defined, and why ACLs change what users can see or edit.

What a table is

A table is the structure that stores records in ServiceNow. Every task, incident, user, CI, or custom app record lives in a table.

  • task is the parent table for work items.
  • incident extends task and adds incident-specific fields.
  • sys_user stores user records and is used by assignment, roles, approvals, and audit trails.
  • cmdb_ci is the base table for configuration items in the CMDB.

How tables are used

  1. Determine the business object you need to store.
  2. Create or choose the right parent table.
  3. Add fields in the dictionary with clear names and labels.
  4. Control visibility and edit access with ACLs.
  5. Expose the data through forms, lists, reports, flows, and APIs.

Dictionary and fields

The dictionary defines the fields on a table, their data types, defaults, reference targets, and whether they are inherited from a parent table.

// Example table inspection
var gr = new GlideRecord('sys_db_object');
gr.addQuery('name', 'incident');
gr.query();
if (gr.next()) {
  gs.info('Incident table label: ' + gr.getValue('label'));

Examples you should know

  • incident for service desk cases.
  • problem for root-cause work.
  • change_request for controlled change.
  • sc_req_item for catalog fulfillment.
  • sys_attachment for files and documents.
  • sys_security_acl for access control rules.

ACLs in plain language

ACLs decide who can read, create, write, delete, or execute data and scripts. If a user cannot see a record, update a field, or access an API, an ACL is one of the first things to check.

  1. Check whether the user has the required role.
  2. Check whether the table ACL or field ACL exists.
  3. Check record conditions and script logic.
  4. Check inheritance from parent tables.
  5. Verify the behavior as the affected user, not just as admin.

Common problems

  • Users can open a list but not a record because field ACLs block the form.
  • Records exist but do not appear because the list filter or domain policy is too strict.
  • Custom fields do not show because the dictionary entry was created on the wrong table.
  • Data looks duplicated because the table hierarchy was not planned before extension.
  • Admin sees data that a normal user cannot, because ACLs are bypassed for elevated roles.

Practice task

Open incident.list, inspect a record, identify the parent table, check the fields on the form, and then review the ACLs that govern the number, caller, assignment group, and state fields.

Try the same workflow on sys_user and cmdb_ci so you can see how table purpose changes the design and access rules.
